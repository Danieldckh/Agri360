### Blocking ❌

1. **[ui/js/app.js:516] `TypeError: Cannot read properties of null (reading 'firstChild')`** — null-deref in the navigation path. Fires on every `navigateToPage()` call. Does not prevent rendering but pollutes the console and indicates a missing guard in the transition logic. Expected: the code at line 516 reads `someElement.firstChild` where `someElement` is null — probably a container lookup that missed. Fix: guard the null case or fix the selector.

2. **[pages/client-list/client-list.js, ui/js/app.js] `client-list` page is not reachable.** `window.renderClientListPage` is `undefined` at runtime. The existing `client-list.js` was built with the `insertTemplate()` pattern (not the `renderXxxPage` export pattern). The Ch. 7 frontend work extended the file with lifecycle dropdown + search + row nav, but never exposed `window.renderClientListPage` and never registered `client-list` in `pageRenderers` in `ui/js/app.js`. Calling `window.navigateToPage('client-list')` is a silent no-op — content area stays on the previously-selected department (Design/Magazine by default). Fix: either (a) convert `client-list.js` to export `window.renderClientListPage` and register in `pageRenderers`, or (b) extend `navigateToPage` to recognize the existing `insertTemplate` pattern for `client-list`.

3. **[ui/js/app.js, pages/*] No visible entry point to `client-list` in the real CRM layout.** The live CRM's single sidebar contains these nav items and NOTHING else: `Design`, `Content Calendars`, `Agri for All`, `Magazine`, `Web Design`, `Banners`, `Proposals`. The top bar has only the Agri360 logo and a User menu (Edit Profile + Sign Out). There is NO "Clients" nav item anywhere, NO top-level section switcher, and NO route by hash/URL that a user could discover. The Ch. 7 plan assumed a "primary sidebar" with Clients/Employees/etc., but the existing CRM never had that layer — the existing sidebar is already the primary (and currently-only) sidebar. Fix: add a "Clients" nav item to the existing `.sidebar` nav list so users can actually reach the client-list page. Coordinate with the upcoming Ch. 12 department-queues work so that the sidebar grouping remains sane (e.g. a "CRM" group above the departments group containing Clients, Employees, Messaging, Admin; and the Design/Content Calendars/Magazine items stay as a separate "Production" or "Departments" group).
